## Bucket 2: Remaining -- Transitive, patchable with upstream bumps

### `rsa` 0.9.10 -- RUSTSEC-2023-0071 (Marvin Attack timing sidechannel)

- **Status:** No fixed version available upstream.
- **Path:** `sqlx-mysql 0.8.6` -> `rsa 0.9.10`
- **Impact:** Low. This project uses PostgreSQL only. The `rsa` crate is pulled in by the
  `sqlx-mysql` compile-time feature (used by sqlx-macros for proc-macro compilation) but
  is never used at runtime. The `rsa` crate is not in the actual dependency tree for our
  build targets (confirmed via `cargo tree --target all -i rsa` showing nothing).
- **Action:** No action possible. Wait for `rsa` crate maintainers to release a fix.

### `rustls-pemfile` 2.2.0 -- RUSTSEC-2025-0134 (unmaintained)

- **Status:** Pulled in by `tonic 0.12.3` (used by google-cloud-kms).
- **Path:** `tonic 0.12.3` -> `rustls-pemfile 2.2.0`
- **Impact:** Low. The pemfile parser is used for TLS cert loading in gRPC, not directly
  by our code.
- **Action:** Wait for `tonic` to release a version that drops `rustls-pemfile` dependency
  (tracked in tonic upstream).

### `dotenv` 0.15.0 -- RUSTSEC-2021-0141 (unmaintained)

- **Status:** Direct dependency. Can be replaced with `dotenvy` (maintained fork).
- **Path:** Direct workspace dependency.
- **Impact:** Low. The crate works fine, it's just unmaintained.
- **Action:** Planned replacement with `dotenvy` in a separate PR to avoid scope creep
  in this security triage.

## Bucket 3: Upstream-blocked -- `ucan` / `did-key` ecosystem

The `ucan` (0.7.0-alpha.1) and `did-key` (0.2.1) crates are the latest published versions
on crates.io. Both are unmaintained and pull in a chain of outdated/vulnerable dependencies:

### Vulnerabilities via `did-key`:

| Crate | Advisory | Notes |
|---|---|---|
| `curve25519-dalek` 3.2.0 | RUSTSEC-2024-0344 (timing variability) | Needs >=4.1.3, but `did-key` pins v3 |
| `ed25519-dalek` 1.0.1 | RUSTSEC-2022-0093 (double pubkey oracle attack) | Needs >=2, but `did-key` pins v1 |

### Unmaintained crates via `ucan` / `did-key`:

| Crate | Advisory | Pulled by |
|---|---|---|
| `async-std` 1.13.2 | RUSTSEC-2025-0052 (discontinued) | `ucan` |
| `atomic-polyfill` 1.0.3 | RUSTSEC-2023-0089 (unmaintained) | `did-key` -> `bls12_381_plus` -> `heapless` |
| `instant` 0.1.13 | RUSTSEC-2024-0384 (unmaintained) | `ucan`, `nostr` |
| `proc-macro-error` 1.0.4 | RUSTSEC-2024-0370 (unmaintained) | `ucan` -> `multihash-derive` |
| `stdweb` 0.4.20 | RUSTSEC-2020-0056 (unmaintained) | `instant` (transitive) |

### Assessment

The `ucan` and `did-key` crates have no newer releases. The `curve25519-dalek` and
`ed25519-dalek` vulnerabilities are real but constrained:

- `curve25519-dalek` timing issue affects scalar subtraction -- relevant for signing
  operations but low practical risk for our UCAN verification use case.
- `ed25519-dalek` double-pubkey oracle requires attacker to control both public keys in
  a dual-key signing scenario -- not applicable to standard UCAN flows.

### Decision

**Action:** Accept upstream-blocked status for now. Vendor the minimal UCAN verification
logic needed (removing the `ucan` and `did-key` dependencies entirely) as a dedicated
follow-up branch. This is preferred over forking `did-key` because our usage surface is
small and vendoring eliminates the entire unmaintained dependency chain.

Note: The `instant` crate is also pulled by `nostr` 0.44.x and `nostr` 0.37.x. This will
likely be resolved when nostr-sdk releases a version that drops the `instant` dependency.
